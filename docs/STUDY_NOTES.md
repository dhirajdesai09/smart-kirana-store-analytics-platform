# StorePulse Study Notes

These notes explain the exact functionality of the project in a revision-friendly way.
They describe the MySQL-backed version verified on July 4, 2026.

## 1. One-Line Explanation

StorePulse is a SaaS dashboard that helps kirana store owners manage products, track sales, update stock automatically, analyze business performance, forecast demand, and export reports.

## 2. Complete User Flow

1. Owner registers a store.
2. Owner logs in.
3. Owner creates categories and suppliers.
4. Owner adds products with SKU, price, stock, reorder level, and expiry date.
5. Cashier records customer sales from the POS page.
6. Backend generates invoice and sale items.
7. Backend reduces product stock automatically.
8. Backend records inventory logs.
9. Dashboard displays revenue, profit, top products, alerts, and low stock.
10. Analytics page shows trends, category mix, basket analysis, forecast, and festival spikes.
11. Reports page exports business data as CSV, Excel, or PDF.

## 3. Authentication Flow

Register:

- User enters email, password, name, and store details.
- Backend creates user, store, owner membership, and profile.

Login:

- Frontend sends username and password.
- Backend returns access token and refresh token.
- Frontend stores tokens in local storage.

Protected APIs:

- Axios sends `Authorization: Bearer <access_token>`.
- If access token expires, refresh token gets a new access token.

## 4. Inventory Functionality

Inventory contains:

- Product name
- SKU
- Barcode
- Category
- Supplier
- Quantity
- Reorder level
- Reorder quantity
- Purchase price
- Selling price
- Tax rate
- Expiry date

Important logic:

- Low stock: `quantity <= reorder_level`
- Stock value: `quantity * purchase_price`
- Profit margin: `(selling_price - purchase_price) / selling_price * 100`
- Restock recommendation: reorder quantity or enough stock to reach above reorder level

## 5. Sales Functionality

The Sales page works like a POS screen.

Cashier can:

- Search product
- Add product to cart
- Change quantity
- Add discount
- Select payment mode
- Complete sale

Backend then:

- Validates product belongs to same store
- Checks stock availability
- Creates invoice number
- Creates `Sale`
- Creates multiple `SaleItem` rows
- Calculates subtotal, discount, GST, grand total, and profit
- Reduces product stock
- Adds inventory log

## 7. Analytics Functionality

Products with low sales but high stock value are considered slow moving.

Use case:

- Owner can reduce purchase of slow products.
- Owner can run offers to clear dead stock.

### Category Analysis

Groups sales by product category.

Use case:

- Understand which category gives more revenue.
- Compare snacks, dairy, rice, oil, festive products, etc.

- Combo offers
- Cross-selling

Uses recent daily sales to predict next-week demand.

It checks:

- Average daily sales
- Recent trend

Output:

- Next week demand
- Recommended restock

## 10. Database Study Notes

### Why MySQL Is Used

StorePulse data is relational and transaction-heavy. Products belong to stores
and categories, invoices contain multiple sale items, and checkout updates
several related records together. MySQL provides foreign keys, unique and check
constraints, indexes, transactions, and row locking for this workflow.

The configured connection is:

```text
Engine: django.db.backends.mysql
Database: storepulse
Host: 127.0.0.1
Port: 3306
Character set: utf8mb4
```

### All 20 Tables To Remember

StorePulse domain tables:

1. `accounts_store`: business/store details and owner.
2. `accounts_staffmembership`: user-to-store role mapping.
3. `accounts_userprofile`: phone, timezone, avatar, and default store.
4. `inventory_category`: store-specific product categories.
5. `inventory_supplier`: supplier and lead-time details.
6. `inventory_product`: catalog, prices, tax, expiry, and current quantity.
7. `inventory_inventorylog`: stock movement and audit history.
8. `sales_sale`: invoice header and totals.
9. `sales_saleitem`: invoice lines and price/profit snapshots.
10. `reports_analyticsreport`: report/export generation history.

Django framework tables:

11. `auth_user`: users and hashed passwords.
12. `auth_group`: permission groups.
13. `auth_permission`: model permissions.
14. `auth_group_permissions`: group-permission bridge.
15. `auth_user_groups`: user-group bridge.
16. `auth_user_user_permissions`: direct user permissions.
17. `django_content_type`: installed model identities.
18. `django_migrations`: applied schema migration history.
19. `django_session`: Django Admin sessions.
20. `django_admin_log`: Django Admin action history.

### Most Important Foreign Keys

```text
Store.owner -> User
StaffMembership.user -> User
StaffMembership.store -> Store
UserProfile.user -> User
UserProfile.default_store -> Store
Category.store -> Store
Supplier.store -> Store
Product.store -> Store
Product.category -> Category
Product.supplier -> Supplier
InventoryLog.product -> Product
Sale.store -> Store
SaleItem.sale -> Sale
SaleItem.product -> Product
AnalyticsReport.store -> Store
```

### Normalization

- Store details are stored once in `accounts_store`.
- Category and supplier data are not repeated in each invoice.
- Invoice headers and lines are separated into `sales_sale` and
  `sales_saleitem`, representing a one-to-many relationship.
- Staff roles use a bridge table because a user can belong to multiple stores.
- Historical sale values are deliberately duplicated as snapshots. This is
  controlled denormalization for audit accuracy.

### Constraints

- `(store, sku)` is unique for products.
- `(store, name)` is unique for categories and suppliers.
- `(store, invoice_number)` is unique for sales.
- `(user, store)` is unique for memberships.
- Product purchase and selling prices must be zero or positive.

### Delete Behavior

- `CASCADE`: deleting a store removes its owned business records.
- `PROTECT`: categories and sold products cannot be removed when their history
  still depends on them.
- `SET_NULL`: deleting a supplier or creator keeps historical records.

### Transactions and Row Locking

`transaction.atomic()` guarantees all-or-nothing checkout. `select_for_update()`
locks product rows while stock is deducted, reducing the risk of two cashiers
selling the same final units concurrently.

### ORM-to-Pandas Flow

1. Django ORM filters completed sales for the authenticated store.
2. `.values()` returns only analytics fields.
3. Pandas builds a DataFrame.
4. Decimal columns become numeric values.
5. Sale timestamps are converted to the configured India timezone.
6. Grouping, merging, pivots, combinations, and NumPy calculations generate
   API-ready results.

## 13. How To Explain This Project In 60 Seconds

StorePulse is a full-stack analytics platform for kirana stores. The owner can manage inventory, suppliers, and product categories, while cashiers can record POS sales. Every sale automatically generates invoice totals, updates stock, and creates inventory logs. The backend uses Django REST Framework with JWT authentication and store-scoped permissions. The analytics engine uses Pandas and NumPy to calculate KPIs, sales trends, product performance, basket analysis, demand forecasts, festival sales spikes, and alerts. The React frontend shows these insights in a SaaS-style dashboard with charts, filters, reports, and staff management. The project also includes Docker support and deployment-ready configuration.

## 15. Limitations And Future Scope

Current limitations:

- Forecasting is simple moving-average/trend based.
- Festival dates are currently maintained in code for 2025 and 2026.
- Payment gateway is not integrated.
- Password reset currently returns a safe response but needs a production email backend and reset-token flow.
- Email/SMS/WhatsApp alerts are not fully implemented.
- Barcode scanning UI can be expanded.
- The frontend production bundle can be further code-split.

Future scope:

- Real-time WebSocket dashboard
- Barcode camera scanning
- OCR bill scanning
- WhatsApp notifications
- AI-generated business insights
- Multi-store owner dashboard
- Advanced ML forecasting
- Supplier purchase order automation
